
  2. Focus change triggers a full redraw even when only borders need repainting

  compositor/mod.rs set_focus()

  The only visual change from a focus switch is the border colour of two windows. The current code calls mark_full_redraw()
  unconditionally. Instead, compute 4 border-strip rects for the old focused window and 4 for the new one, union them, and
  call mark_damage. Clicking between windows would stop rebuilding the entire back-buffer.

  3. blit_to_back_uniform per-pixel blend can do two channels per multiply

  compositor/render.rs:150–160

  The inactive-dimming path blends three channels with three separate multiplies + shifts per pixel. A standard trick
  processes two channels in one multiply: (px & 0x00FF00FF) * alpha folds R and B together since they don't occupy adjacent
  bytes. Worth doing if dimming is used heavily. Requires the pixel format to have at most one channel in bits 16–23 and 0–7,
  which is true for the common 0xXXRRGGBB layout.

  6. Floating drag doesn't enforce a minimum on-screen margin

  compositor/mod.rs update_drag() → MoveFloating

  The x/y clamp only prevents the window origin from leaving the screen (clamp(0, width-1)). A window can be dragged so its
  body is entirely off-screen with only its top-left pixel visible. Clamping to (-(w - MIN_VISIBLE), screen_w - MIN_VISIBLE)
  where MIN_VISIBLE = 64 would keep it reachable.

  10. Fullscreen toggle

  A ShortcutAction::Fullscreen that temporarily gives the focused tiled window the full available area (saving and restoring
  its ratio). One keybind (e.g. Super+F) would remove the need to adjust panels manually and is the most-requested tiling
  feature. Implementation: set a fullscreen_window: Option<WindowId> field; when set, recalculate_toplevel_layout skips all
  other tiled windows.


  11. Cursor capture / hiding for games

  No API to hide the cursor or lock it to a window (needed for FPS-style mouse look). A window event or message like
  SetCursorMode { hidden: bool, locked: bool } would cover the common cases; the compositor would suppress rendering and
  instead accumulate raw deltas for the locked window.

  2. BUGS, EDGE CASES, AND MISSING ERROR HANDLING

       CRITICAL ISSUES

       2. Global Allocator No Bounds Checking (kernel/core/src/memory/global_allocator.rs:25-53)
         - Issue: init() calls .unwrap() if no USABLE entry has ≥4 MiB, no fallback
         - Line: 33-34: finds first entry with entry.length >= GLOBAL_ALLOCATOR_SIZE then unwraps
         - Risk: Panic if Limine memory map has no suitable range (should fail gracefully)

       3. User Pointer Validation Incomplete (kernel/core/src/syscall_handlers/mod.rs:27-50)
         - Issue: validate_user_ptr() checks bounds but doesn't verify allocation in user_vaddr_set
         - Line: 48-50: calls is_user_vaddr_valid_range() only after acquiring run_queue lock, but vaddr_set can be
       modified concurrently
         - Risk: TOCTOU race if user task modifies its own page table while syscall validates
         - Impact: Potential out-of-bounds memory access via syscall
       4. Context Switch CR3 Switch Not Atomic w.r.t. Interrupts (kernel/core/src/task/local_scheduler.rs:127-131)
         - Issue: Cr3::write() in schedule_from_interrupt() isn't wrapped in atomic context
         - Line: 127-131: CR3 switch happens during interrupt handling with no special synchronization
         - Risk: If a nested interrupt fires between CR3 write and TLB flush on KVM, stale TLB entries could cause #PF or
        worse
         - Impact: Rare race on very high-latency systems (real hardware likely OK due to microcode)
       5. No Free Page Table Frames After User Task Cleanup (kernel/core/src/task/task.rs:138-193)
         - Issue: free_user_address_space() walks page tables and calls free_frame() for data frames but some test
       configs never free the task
         - Line: 173-192: frees L1/L2/L3 frames but only if they're already in the map
         - Risk: In tests that create user tasks but never exit them (e.g., test_user_task_runs), frames accumulate → OOM
         - Impact: Tests with many task spawns fail with allocator exhaustion
       6. sys_exit Never Enables Interrupts Post-Unlock (kernel/core/src/syscall_handlers/task.rs:11-60)
         - Issue: sys_exit() calls interrupts::enable() then hlt(), but if it re-enters the scheduler, no safety
         - Line: 56-58: enables interrupts then loops in hlt forever
         - Risk: Low; hlt in a loop is safe but non-standard (other CPUs won't reschedule this one)
         - Impact: Exiting task never scheduled again (expected behavior)

       HIGH-PRIORITY ISSUES

       7. KVM SS.RPL Stripping Workaround May Fail on Real Hardware (kernel/core/src/interrupt/handlers.rs:205-214)
         - Issue: Code patches SS RPL to 3 if returning to ring 3, assuming KVM bug
         - Line: 212-214: or rax, 3 to force RPL=3
         - Risk: Real hardware may reject SS=0x1B (RPL in segment limit check)
         - Impact: GPF on non-KVM x86_64 systems is possible but unlikely (most allow flexible SS)
       8. IDE Disk Driver Infinite Loop on Error (kernel/core/src/drivers/disk.rs:79-92)
         - Issue: init() loops waiting for DRQ but has no timeout
         - Line: 87-92: loop with no maximum iterations
         - Risk: Broken/slow IDE controllers cause hang during init
         - Impact: Kernel hangs at boot (no timeout, no async abort)
       9. Mouse/Keyboard Buffers Silently Drop on Overflow (kernel/core/src/drivers/keyboard.rs:44-51, mouse.rs:24-30)
         - Issue: Both buffers only push if count < BUFFER_SIZE, otherwise drop
         - Line: 44-50 (kbd), 25-29 (mouse): if self.count < BUFFER_SIZE
         - Risk: Rapid key/mouse input can lose events without notification
         - Impact: Dropped input; user won't know what happened
       10. No Validation of ELF Load Address Against User Space (kernel/core/src/user_task_from_elf.rs:76-X)
         - Issue: ELF segments are mapped without checking if they fit in user space or overlap
         - Risk: Malformed ELF with segments at 0xffffffffffffff could cause load failure silently
         - Impact: Failed task spawn returns 0, user sees no error

       MEDIUM-PRIORITY ISSUES

       12. Raw Pointer Dereference Without Alignment Checks (kernel/core/src/interrupt/handlers.rs:57-68)
         - Issue: General protection fault handler reads iretq frame via raw pointer
         - Line: 57-68: unsafe { let ptr = rsp as *const u64; let iretq_rip = *ptr; ... }
         - Risk: If RSP is corrupted, dereference causes recursive #GPF → double fault
         - Impact: Unhelpful panic message if GPF was due to stack corruption
       13. No Maximum ELF Size Validation (kernel/core/src/syscall_handlers/task.rs:83-85)
         - Issue: sys_spawn() checks elf_len > 64 MiB but doesn't validate against available memory
         - Line: 84: if elf_len == 0 || elf_len > 64 * 1024 * 1024
         - Risk: Accepting an ELF doesn't guarantee parse/load won't fail later
         - Impact: Spawn fails partway through → partial task state left
       15. No Deadlock Detection in IPC Blocking (kernel/core/src/ipc/mod.rs)
         - Issue: Channels can deadlock if two tasks try to send on full channels to each other
         - Risk: No cycle detection in waiter graph
         - Impact: Tasks hang forever (expected behavior in OS, but no detection)

       16. Unused Legacy Context Switch Code (kernel/core/src/task/context.rs:5-147)

       17. No Bounds Check on name_len in Syscalls (kernel/core/src/syscall_handlers/task.rs:105-109)

       5. STABILITY & RELIABILITY CONCERNS

       CRASH VECTORS

       1. Panic in Interrupt Handler - timer_interrupt_handler_inner can panic (line 143-147 debug assertion) if context
       is corrupt
       2. Stack Overflow Not Always Caught - While guard pages exist, some nested interrupt scenarios might bypass them
       3. User Memory Corruption - syscall handlers read from user ptrs without retry-on-fault logic
       4. Disk I/O Deadlock - IDE driver polls forever on hardware error

       RACE CONDITIONS

       1. TOCTOU in Syscall Validation - User task can modify page table while syscall checks it (low probability but
       possible)
       2. Nested Interrupt CR3 Switch - KVM migration between CR3 write and TLB flush (very low probability)
       3. Task State Machine Transitions - State changes are atomic but watchers might miss transitions (e.g.,
       Zombie→removed from table)

       RELIABILITY NOTES

       - Strengths: Excellent use of atomic types, spin::Mutex for critical sections, clear ownership via Arc
       - Weaknesses: No fault recovery, aggressive unwrap()s in init paths, silent failures in I/O drivers
       - Testing: Good test coverage in kernel/tests/ but doesn't cover fault injection scenarios

  Stability / Correctness

  3. Global allocator panics on exotic memory maps

  kernel/core/src/memory/global_allocator.rs — init() does .unwrap() on finding a 4 MiB USABLE Limine entry. If the map is
  fragmented (no single 4 MiB chunk) it panics. Should try smaller sizes or collect pages.

  4. ELF segments never validated against user address space

  kernel/core/src/user_task_from_elf.rs — a crafted ELF could specify p_vaddr in the kernel half (0xffff...), or overlapping
  segments. The current code calls mapper.map_to() and .unwrap()s — this panics the kernel rather than returning
  SpawnError::InvalidElf. The check should be: p_vaddr + p_memsz < LOWER_HALF_END and no overlap between LOAD segments.
